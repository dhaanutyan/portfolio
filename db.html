Database manager for merging and comparing

Разработка утилиты которая осуществляет слияние 2х баз данных, сохраняет этот процесс в временный файл в котором можно его корректировать, сохранять и продолжать корректирование что бы затем экспортировать слитую базу данных в нужный формат.

The problem to think and to solve

Пытаемся понять какого типа ошибки могут возникать при слиянии. Раскладываем базу на таблицы. Перебираем различные варианты не совпадений параметров. — Для различий в таблицах в виде отсутствия записи или параметра решений не нужно так как чаще всего по умолчанию при слиянии баз сохраняется максимально возможное количество информации. — Для различий в таблицах в виде когда информация записывается поверх существующей и находит на этом месте другую информацию возникнет конфликт. Предполагаемо их и нужно решать. — При слиянии баз данных обычно прописывают правила и условия слияния. — Для одной записи могут быть несовпадения по нескольким параметрам. — Для записи типа один к одному можно предложить варианты ручного исправления в виде использования данных А, использования данных Б и заполнение поля произвольным значением. — Для записи типа один ко многим кроме вариантов А и Б можно предложить выбор из параметров. Для понимания процесса слияния баз данных представим себе определенную структуру базы данных для примера:

Research

Собираем тест группу из программистов имеющих опыт работы с базами. Расспрашиваем подробно чем они пользуются для этого и как выглядит их процесс. Какие чаще всего возникают ошибки и как они их исправляют. В том числе собираем скриншоты интерфейсов связанных с базами данных сравнением двух баз данных и слиянием данных.

User flow

На основе проведенных интервью и собранных данных воссоздаем процесс использования утилиты.

First try

Пытаемся представить себе вид этой утилиты в трех экранах. Предполагаю что процесс может быть пошаговым. Предполагаем что — нужно подгрузить 2 базы данных и куда то сохранить третью. Так же возможно что базы данных будут загружаться с удаленных серверов. — Нужно предложить правила и настройки применимые для всей базы перед слиянием. — Хорошо бы держать перед глазами структуру загруженных баз ( предполагаем что она одинаковая или вроде того). — Нужно оставить возможность поштучного редактирования конфликтов.

Testing

Возвращаемся к тест группе программистов. Предлагаем им оценить прототип утилиты для слияния баз и исправления ошибок. Собираем обширный фидбэк: — Программистам хотелось бы наглядно видеть количество конфликтов в каждой таблице отдельно. Так как от этого может зависеть тип конфликта. — Правила действующие автоматически хотелось бы применять к каждой таблице отдельно как и ко всей базе в целом. — Очень не хватает предпросмотра таблицы в выделенными в ней конфликтами. — Открытую запись для редактирования привычнее видеть в столбик а не в строку. — Названия параметров внутри таблицы в левой плашке бесполезны. и не интересны. — Утилита не будет подгружать фотографии из дампов (точно!) — Удаление конфликтных параметров вообще для массового решения вопроса нецелесообразно (см пункт о том что при слиянии пытаются сохранить максимум данных) Собираем новую версию. Учитываем ошибки. Снова тестируем на программистах. Остаются детали. Исправляем. Хотелок еще много, поэтому останавливаюсь на определенном функционале и возможностях.
