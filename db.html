Research

A database is kind of related data and the way it is organised. The relational database help to track information about on example users, their names, logins information, various groups and projects they are involved and normalised it into a user table, groups table and projects table, accordingly.

When information was being collected about a user, information stored in the optional tables would be found by searching for a special key. For instance, if the login name of a user is unique, all information about that user would be recorded with the login name as its key. Keys and table names play a "key" role when comparing or merging databases because it is for them to determine the identity of data.

A relational database management system uses MERGE statements to write new or overwrite existing records depending on whether condition matches. Structure and tables comparing before merging. Each time software or script find differences it doesn't know what to do unless you create a rule for each situation or manage it manually. That's why It is much easier merge the databases with the same structure, especially if you try to automate the process. Lucky I have this particular case!

In an attempt to understand what kind of errors can occur till merging I decompose the database into tables and search for various situations with unequal data.

— If the one field in the database A matches the same field in the database B

— If the one field in the database A does not match the same field in the database B

— If the one field does not exist in the database A but does exist in the database B.

— If the one field does not exist in the database B but does exist in the database A.

Audience

Developers, data architects, and others who regularly work with databases often need to locate changes, migrate differences, or synchronise versions of database tables. They need database compare and merge tool to satisfy all these requirements.

I gather a test group of 8 programmers who have experience with databases and ask in detail what they use for this and how their process looks. What are the most common mistakes and how do they fix them. Including we collect screenshots of interfaces related to databases by comparing two databases and merging data.

User flow

Based on the interviews and data collected, we recreate the process of using the utility.

First try

Пытаемся представить себе вид этой утилиты в трех экранах. Предполагаю что процесс может быть пошаговым. Предполагаем что — нужно подгрузить 2 базы данных и куда то сохранить третью. Так же возможно что базы данных будут загружаться с удаленных серверов. — Нужно предложить правила и настройки применимые для всей базы перед слиянием. — Хорошо бы держать перед глазами структуру загруженных баз ( предполагаем что она одинаковая или вроде того). — Нужно оставить возможность поштучного редактирования конфликтов.

Testing

Возвращаемся к тест группе программистов. Предлагаем им оценить прототип утилиты для слияния баз и исправления ошибок. Собираем обширный фидбэк: — Программистам хотелось бы наглядно видеть количество конфликтов в каждой таблице отдельно. Так как от этого может зависеть тип конфликта. — Правила действующие автоматически хотелось бы применять к каждой таблице отдельно как и ко всей базе в целом. — Очень не хватает предпросмотра таблицы в выделенными в ней конфликтами. — Открытую запись для редактирования привычнее видеть в столбик а не в строку. — Названия параметров внутри таблицы в левой плашке бесполезны. и не интересны. — Утилита не будет подгружать фотографии из дампов (точно!) — Удаление конфликтных параметров вообще для массового решения вопроса нецелесообразно (см пункт о том что при слиянии пытаются сохранить максимум данных) Собираем новую версию. Учитываем ошибки. Снова тестируем на программистах. Остаются детали. Исправляем. Хотелок еще много, поэтому останавливаюсь на определенном функционале и возможностях.
